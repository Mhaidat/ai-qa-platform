# Architecture

## 1. Core Domain Model

The AI QA Platform is built around the following core business entities.

---

## Organization

Represents a customer company using the platform.

Examples:

- TestCrew
- ABC Software
- Digital Transformation Company

Main responsibility:

- Owns users, projects, subscriptions, and settings.

---

## User

Represents a person using the platform.

A user belongs to an Organization and may have different roles.

Main roles:

- Organization Owner
- Administrator
- Business Analyst
- Manual QA Engineer
- Automation QA Engineer
- QA Lead

---

## Project

Represents a software project inside an Organization.

Examples:

- Mining Licensing Platform
- Banking Mobile App
- E-Commerce Website

A project contains Stories, Knowledge Base, Test Cases, Jira integration, and Automation outputs.

---

## Story

Represents a software requirement or user story.

A Story can be:

- Imported from Jira
- Created manually

Stories are the core requirement entity in the platform.

Epics, Releases, and Sprints are optional.

---

## Knowledge Base

Represents the project understanding extracted from Stories.

The Knowledge Base helps AI analyze requirements using project context instead of analyzing each Story in isolation.

Initial knowledge items:

- Business Rules
- Gaps
- Ambiguities
- Suggested Acceptance Criteria
- Potential Conflicts

---

## AI Analysis

Represents the result of AI analysis for a Story.

It may include:

- Extracted business rules
- Missing information
- Ambiguous statements
- Suggested acceptance criteria
- Related stories
- Possible conflicts

---

## Test Case

Represents a manual test case generated by AI or created manually by QA.

A Test Case belongs to a Story.

A Test Case can be:

- AI Generated
- Manually Created
- Edited by Manual QA
- Approved
- Linked to Jira
- Used for Automation

---

## Jira Connection

Represents the integration between the platform and a Jira instance.

Used for:

- Importing Stories
- Linking Test Cases
- Manual synchronization

Jira synchronization is always user-initiated.

---

## Automation Job

Represents a Playwright generation request.

Automation is generated only from approved Test Cases.

The output may include:

- Playwright TypeScript tests
- Page Object Model
- Assertions
- Test data
- Downloadable ZIP project

---
